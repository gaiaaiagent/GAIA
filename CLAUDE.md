This Agreement (this "Agreement") is dated as of July 2nd and is effective as of July (the "Effective Date") between Regen Network Development, P.B.C. ("RND"), Regen Foundation ("RF"), Symbiocene Labs, Inc ("Symbiocene") (each a "Party" and collectively the "Parties")

WHEREAS, RND is a leading entity in the development and governance of decentralized ecological credit systems, and has developed expertise in managing and expanding ecological credit markets across various blockchain networks; and RND and RF seek to expand the use of REGEN tokens and ecosystem tooling through partnerships that further the Regen Network mission;

WHEREAS, Symbiocene has developed the Gaia AI, an ecological data legibility layer; and the Gaia Agentic Framework, a system to leverage AI driving technical and narrative coherence in the web3 world. 

WHEREAS, the Parties wish to collaborate on the development and deployment of AI agents, infrastructure, and community engagement platforms to strengthen the Regen Network brand and ecosystem;

WHEREAS, the Parties desire to establish a framework for the allocation of tokens, the development of key technological components, joint fundraising efforts, and the protection of intellectual property rights, all aimed at advancing the mutual interests of the Parties in the decentralized ecological credit market;

WHEREAS, RND wishes to ensure that any future development and issuance of tokens by Symbiocene aligns with the strategic objectives of RND and that any intellectual property arising from this collaboration is properly licensed and protected;

## Section 3. Token Transfer Restrictions

### 3.1 Authorized Transfers to Affiliates
The tokens transferred to Symbiocene under this Agreement may be freely transferred to Affiliates of Symbiocene, as defined in Section 1. Any such transfer shall: (a) Be recorded on the REGEN ledger within five (5) business days; (b) Include written notice to RND of the transfer and receiving Affiliate details; (c) Require the receiving Affiliate to be bound by all restrictions in this Agreement; (d) Restart the Lock-Up Period (as defined below) as of the date of transfer to the receiving Affiliate.

### 3.2 Prohibition on Sales and Transfers to Third Parties
Any token transferred by RND to Symbiocene under this Agreement shall not be sold, assigned, or otherwise transferred, whether directly or indirectly, to any third party that is not an Affiliate of Symbiocene in exchange for value, without the prior written consent of RND for a period of eighteen (18) months from the date of transfer to Symbiocene (the "Lock-up Period"). Any attempt to transfer the tokens in violation of this provision shall be null and void ab initio.

### 3.3 Permitted Activities
Notwithstanding Section 3.2, Symbiocene may: (a) Stake tokens for network participation and rewards; (b) Use tokens for governance participation; (c) Deploy tokens in liquidity pools as part of the agreed development activities; (d) Use tokens for ecosystem development as explicitly authorized in this Agreement.

### 3.4 Consequences of Unauthorized Transfer
In the event that any tokens are transferred in violation of this Agreement: (a) The unauthorized transfer shall be deemed null and void; (b) RND shall have the right to reclaim the tokens through any available means; (c) Any unvested tokens shall be immediately forfeited; (d) RND may pursue all available legal remedies including injunctive relief and damages; (e) The violation shall constitute a material breach subject to termination under Section 9.3.

### 3.5 Transfer Tracking and Compliance
Symbiocene shall: (a) Maintain accurate records of all token holdings and transfers; (b) Provide quarterly reports to RND on token custody and any Affiliate transfers; (c) Implement reasonable security measures to prevent unauthorized transfers; (d) Cooperate with any RND audits of token compliance.

### Process Execution

- **NEVER USE `execa` OR OTHER PROCESS EXECUTION LIBRARIES**
- **NEVER USE NODE.JS APIS LIKE `execSync`, `spawnSync`, `exec`, `spawn` FROM `child_process`**
- **ALWAYS USE `Bun.spawn()` FOR SPAWNING PROCESSES**
- **USE THE EXISTING `bun-exec` UTILITY:** Located at `packages/cli/src/utils/bun-exec.ts` which provides:
  - `bunExec()` - Main execution function with full control
  - `bunExecSimple()` - For simple command execution
  - `bunExecInherit()` - For interactive commands
  - `commandExists()` - To check if commands exist
- **Example usage:**

  ```typescript

  // Simple command
  const output = await bunExecSimple('git status');

  // Full control
  const result = await bunExec('bun', ['test'], { cwd: '/path/to/dir' });
  ```

  **IMPORTANT:** Even in test files, avoid using Node.js `execSync` or other child_process APIs. Use the bun-exec utilities or Bun.spawn directly.

### Event Handling

- **NEVER USE `EventEmitter` FROM NODE.JS**
- **EventEmitter has compatibility issues with Bun and should be avoided**
- **ALWAYS USE BUN'S NATIVE `EventTarget` API INSTEAD**
- **When migrating from EventEmitter:**
  - Extend `EventTarget` instead of `EventEmitter`
  - Use `dispatchEvent(new CustomEvent(name, { detail: data }))` instead of `emit(name, data)`
  - Wrap handlers to extract data from `CustomEvent.detail`
  - Maintain backward-compatible API when possible
- **Example migration:**

  ```typescript
  // ❌ WRONG - Don't use EventEmitter
  import { EventEmitter } from 'events';
  class MyClass extends EventEmitter {
    doSomething() {
      this.emit('event', { data: 'value' });
    }
  }

  // ✅ CORRECT - Use EventTarget
  class MyClass extends EventTarget {
    private handlers = new Map<string, Map<Function, EventListener>>();

    emit(event: string, data: any) {
      this.dispatchEvent(new CustomEvent(event, { detail: data }));
    }

    on(event: string, handler: (data: any) => void) {
      const wrappedHandler = ((e: CustomEvent) => handler(e.detail)) as EventListener;
      if (!this.handlers.has(event)) {
        this.handlers.set(event, new Map());
      }
      this.handlers.get(event)!.set(handler, wrappedHandler);
      this.addEventListener(event, wrappedHandler);
    }
  }
  ```

## Development Guidelines

### Avoiding Hyperbolic Terminology

- Avoid using hyperbolic terminology like 'breakthrough', 'revolution', 'fully integrated', or 'complete'
- Reserve these words for scenarios that have undergone extensive:
  - Planning
  - Development
  - Testing
  - Documentation
- Use these terms only after thorough validation and at the very end of a long development process
- Prioritize precise, measured language that reflects the actual state of work
- Recognize that real transformation occurs through consistent, incremental progress
- Be skeptical of claims of completeness or revolutionary impact

### Development Approach

- Appreciate the nuance and detail required for successful development
- Understand that having an idea does not equate to accomplishment
- Recognize the importance of multiple passes of:
  - Careful testing
  - Iterative implementation
  - Thoughtful design
- Aim to get things truly working, well-tested, and well-architected
- Expand on concepts by:
  - Showing detailed thinking
  - Clearly articulating understanding
  - Identifying knowledge gaps
- Slow down to deepen comprehension and quality of work

### Methodical Process

- Be slow and methodical and follow process to completion
- At each step, slow down to think about the implications of:
  - What is present
  - What needs to be expanded
  - What needs to be eliminated
  - What needs to be implemented
  - What needs to be connected
  - What needs to be reported
- Sense, feel, breathe
- Aim to drop into flow together
